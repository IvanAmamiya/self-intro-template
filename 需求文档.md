# 需求文档（具体方案与架构）

## 1. 项目背景
- 做二次元扩列自介条，帮助用户快速生成可发布的「自我介绍页面 + 图片」。

## 2. 项目目标
### 2.1 核心目标
- 用户在网页中填写信息后，实时预览自介页面。
- 支持一键导出图片（用于发布到社交媒体）。
- 页面可跨端展示（移动端优先，同时兼容桌面端）。
- 纯前端实现，无后端依赖，可直接部署到 Vercel。

### 2.2 非目标（当前版本不做）
- 用户注册登录。
- 云端存储、在线分享链接持久化。
- 多人协作编辑。

## 3. 用户与场景
### 3.1 目标用户
- 二次元扩列用户。
- 需要快速生成统一风格自介图的社交媒体用户。

### 3.2 典型流程
1. 进入页面，选择模板。
2. 填写昵称、属性标签、兴趣、联系方式等字段。
3. 实时看到页面预览。
4. 点击“导出图片”，下载 PNG。
5. 将图片发布到社媒。

## 4. 功能方案（V1）
### 4.1 输入区（Form）
- 基础字段：昵称、头像、性别/属性、地区、爱好、雷点、扩列偏好、联系方式。
- 样式字段：主题色、背景图、字体大小、卡片圆角。
- 字段校验：必填校验、长度限制、联系方式格式校验（基础）。

### 4.2 预览区（Preview）
- 与输入实时双向联动（输入即预览）。
- 提供移动端比例预览（9:16）和通用方图预览（1:1）切换。
- 模板化布局（至少 2 套模板）。

### 4.3 导出区（Export）
- 一键导出 PNG。
- 默认高清导出（建议 2x 像素比）。
- 导出前自动隐藏编辑控件，仅导出展示区域。

### 4.4 本地持久化
- 使用 `localStorage` 自动保存最近一次编辑内容。
- 支持“重置为默认值”。

## 5. 技术架构
### 5.1 总体架构
- 架构类型：单页应用（SPA），纯前端。
- 框架：React（推荐 TypeScript）。
- 打包：Vite。
- 部署：Vercel 静态托管。

### 5.2 模块划分
- `App`：页面骨架与布局。
- `FormPanel`：输入表单与校验。
- `TemplateRenderer`：根据数据渲染不同模板。
- `PreviewCanvas`：预览容器与比例切换。
- `ExportService`：导出图片能力（封装调用）。
- `StorageService`：本地存取与默认值管理。

### 5.3 数据流
1. 用户在 `FormPanel` 输入。
2. 全局状态更新（例如 `useReducer` 或 `zustand`）。
3. `TemplateRenderer` 根据状态实时渲染。
4. 点击导出时，`ExportService` 将预览节点转为图片并下载。

### 5.4 关键依赖建议
- 图片导出：`html-to-image`（或 `dom-to-image-more`）。
- 状态管理：React 内置状态（V1 足够）或 `zustand`（可扩展）。
- 样式：CSS Modules 或 Tailwind（二选一，建议 Tailwind 提升开发效率）。

## 8. 兼容性与性能要求
### 8.1 兼容性
- 现代浏览器最新两个主版本（Chrome / Edge / Safari）。
- 移动端主流 WebView 可用。

### 8.2 性能
- 首屏加载目标：< 2.5s（4G 网络）。
- 输入到预览更新延迟：< 100ms。
- 导出图片耗时：< 2s（中等复杂模板）。

## 9. UI/交互原则
- 左侧输入、右侧预览（桌面）；移动端上下布局。
- 全程可见“当前导出比例”。
- 按钮文案明确：`重置`、`保存草稿`、`导出 PNG`。
- 错误提示就地展示，不打断用户操作。

## 10. 里程碑计划
### M1（1-2 天）
- 完成项目初始化（React + Vite + 基础样式）。
- 完成表单与基础预览联动。

### M2（2-3 天）
- 完成模板渲染系统（至少 2 套模板）。
- 完成本地存储与重置功能。

### M3（1-2 天）
- 完成图片导出、兼容性修正、移动端适配。
- 部署到 Vercel 并验收。

## 13. 二次元自介格式调研（检索结论）

### 13.1 常见内容字段（高频）
- 基本身份：昵称 / 称呼、头像、性别（可选）、地区（可选）。
- 社交标签：属性关键词（如“社恐/慢热/同担友好”等）、MBTI（可选）。
- 兴趣圈层：主坑作品、角色推し、游戏偏好、创作方向（画/文/COS/剪辑等）。
- 交友偏好：想扩什么类型（同坑、同城、一起打本、一起看番等）。
- 雷点与边界：明确“可聊/不聊”“是否接受连麦/私聊”等。
- 联系方式：QQ / 微信 / X / 微博 / B站 UID（建议做可选显示）。

### 13.2 常见版式结构（高频）
1. 顶部信息带：头像 + 昵称 + 1~3 个高亮标签。
2. 中部主信息块：
	 - 关于我（短句）
	 - 爱好与坑位（标签化）
	 - 扩列诉求（我想找什么搭子）
3. 底部联系块：社媒账号 + 二维码（可选）+ 备注。

### 13.3 文案风格特征
- 普遍使用“短句 + 标签”的阅读方式，避免大段文字。
- 语气偏友好、轻松，常见格式：
	- `我是谁` → `我喜欢什么` → `我想找什么人` → `怎么联系我`
- 视觉上偏“可爱/清爽/手账感/ins 风”，并强调“人设一致性”。

### 13.4 画布与导出习惯
- 常见输出格式：PNG（便于社媒转发与保真）。
- 常见画幅：竖版长图（移动端阅读友好）+ 方图（多平台兼容）。
- 导出策略建议：默认 2x 分辨率，确保字体与头像细节清晰。

### 13.5 安全与隐私建议
- 联系方式支持“部分脱敏”（如仅展示后 6 位）。
- 提供“仅展示二维码/仅展示用户名”的开关。
- 不建议在模板中鼓励填写真实姓名、手机号、学校等敏感信息。

### 13.6 对本项目的落地要求（新增）
- V1 预置两类模板：
	1) 标签卡片型（偏可爱）
	2) 信息分栏型（偏简洁）
- 输入字段中增加：`扩列诉求`、`雷点/边界`、`主坑/推し`。
- 预览区支持“长图模式（默认）/方图模式（可切换）”。
- 导出区保留 PNG，并在按钮旁提示“适合社媒发布”。

### 13.7 参考来源（检索样本）
- ProfileCraft（芝士扩列条在线工具）：可见字段包含“基本信息/关于我/推し/标签/二维码”等模块。
- B 站“扩列图/属性图/自介条”相关内容：高频出现“属性图模板”“教程”“短视频展示”。
- 漫展扩列卡设计类文章：强调“信息清晰 + 风格统一 + 隐私保护”。

## 14. 可扩展架构要求（面向后续 PR）

### 14.1 目标
- 支持持续增加“多套样式（Theme）+ 多套布局（Template）”，且尽量避免改动核心业务代码。

### 14.2 设计原则
- 模板与主题通过注册表管理，而非硬编码在页面组件中。
- 通过依赖注入（DI）提供注册表实例，组件仅依赖接口能力。
- 后续 PR 以“新增注册项”为主，降低冲突风险。

### 14.3 结构建议
- `TemplateRegistry`：负责模板注册、读取、枚举。
- `ThemeRegistry`：负责主题注册、读取、枚举。
- `DependencyContainer`：统一管理依赖实例。
- `createAppContainer()`：应用级容器装配入口。
- `registerExternalExtensions()`：外部扩展注入点（默认 no-op）。

### 14.4 变更约束
- 新增模板时：提供独立组件 + 注册定义（id、labelKey、component）。
- 新增主题时：提供样式类 + 注册定义（id、labelKey、previewClassName）。
- 不允许在 `FormPanel` / `PreviewCanvas` 中追加模板/主题硬编码分支。

### 14.5 验收补充
- [ ] 新增一套模板不需要修改核心渲染分支。
- [ ] 新增一套主题不需要修改主题选择逻辑。
- [ ] 外部扩展入口可注入并生效。
